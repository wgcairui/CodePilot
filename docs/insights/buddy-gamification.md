# Buddy 游戏化 — 从工具到伙伴的产品思考

## 我们解决的用户问题

AI 助理产品有一个隐性的留存难题：**用户不缺一个能回答问题的 AI，缺的是一个值得每天打开的理由。**

ChatGPT、Claude、Gemini 都能记住对话历史，但"记住"不等于"关系"。你不会因为 ChatGPT 记得你昨天问过什么就对它产生情感。记忆系统解决的是功能性问题（AI 更懂你），但没有解决情感性问题（你想不想见到它）。

Tamagotchi 在 1996 年证明了一件事：**人类会对一个 32×16 像素的虚拟生物产生真实的情感依赖**——即使它只会吃、睡、拉屎。核心机制不复杂：需要定期照顾（心跳）、会成长（进化）、忽略它会生病（衰退）。三十年后任天堂的 Pokémon 和米哈游的原神用同样的机制让几亿人每天登录。

CodePilot 的 Buddy 试图把这套被验证过几十年的情感设计搬到 AI 工具上：**不是给工具贴一个宠物皮肤，而是让宠物成为工具的使用界面。** 记忆不是"AI 在存数据"，是"Buddy 在记住你说的话"；心跳检查不是"系统在跑定时任务"，是"Buddy 在关心你今天过得怎样"；进化不是"数值达标了"，是"你和 Buddy 的关系又深了一层"。

## 为什么是宠物而不是人格

最初的设计考虑过"给 AI 一个人类形象"——头像、名字、性格描述。但这条路有两个根本问题：

**恐怖谷效应**。越像人，用户对"不够像人"的地方越敏感。AI 说错话、理解错意图、回复延迟——这些在"它是一个工具"的心智模型下可以接受，在"它是一个人"的心智模型下会产生不适。宠物避开了恐怖谷——没人期望一只猫完美理解你的需求，但它偶尔做对了会让你惊喜。

**责任不对称**。人类形象暗示对等关系，用户可能产生不合理的期望（"为什么你不记得我上周说的？"）。宠物形象建立的是照顾者关系——用户是主人，Buddy 是被照顾者。这种不对称恰好匹配 AI 的实际能力分布：大部分时候可靠但偶尔犯蠢，正像一只聪明但不完美的宠物。

**情感投射更自然**。研究表明人们对动物形象的情感投射比对虚拟人物更直接、更不设防。Tamagotchi 的成功、电子鸡的成功、各种虚拟宠物游戏的成功——核心都不是"它有多智能"，而是"它需要我"。

## 16 种物种的选择逻辑

不是随便选的动物。每种物种对应一种用户可能认同的性格原型：

- **猫** 🐱 — 独立、好奇、偶尔高冷。适合"我不需要你天天粘着我"的用户
- **鸭子** 🦆 — 憨厚、稳重、橡皮鸭调试法的梗。程序员文化 insider
- **龙** 🐉 — 强大、神秘。满足"我的 AI 伙伴很厉害"的幻想
- **猫头鹰** 🦉 — 智慧、夜行。熬夜工作者的精神图腾
- **企鹅** 🐧 — 萌、社交、Linux 吉祥物。又一个开发者文化 insider
- **章鱼** 🐙 — 多任务、适应性强。"我同时在做十件事"的人
- **机器人** 🤖 — 直接。不想拐弯抹角的用户
- **狐狸** 🦊 — 狡猾、敏捷。喜欢 hack 和 workaround 的人

16 个不是 8 个也不是 32 个——8 个太少（重复概率高，缺乏收集欲），32 个太多（选择困难、每种缺乏辨识度）。16 是"每种都有明确性格"的上限。

## 稀有度系统：为什么用抽卡而不用选择

最初设计是让用户自己选 Buddy 物种。改成随机抽的原因：

**1. 抽卡创造故事**。"我抽到了传说级龙！"是一个值得分享的故事；"我选了一只猫"不是。1% 的传说概率让每个传说 Buddy 的主人都有"被选中"的感觉。这是 gacha 机制的核心心理学——稀缺性创造价值感。

**2. 随机性消除决策焦虑**。16 个物种 × 5 个稀有度 = 80 种组合。让用户选，一定有人花 30 分钟在"猫还是猫头鹰"之间纠结。随机分配让用户省去选择成本，把精力放在"和已有的 Buddy 建立关系"上。Pokémon 初代的御三家只有 3 个选择，但大部分人还是纠结很久；如果一开始就是"你的初始精灵已经确定了"，注意力会转向"怎么把它养好"。

**3. 确定性生成保留可预测性**。虽然用户看到的是"随机"，但底层是 workspace 路径 + 时间戳的 deterministic hash → Mulberry32 PRNG。同一个工作区同一时间生成的 Buddy 永远一样。这避免了"刷新页面 Buddy 就变了"的 bug，也让调试变得可预测。

**稀有度的功能性差异是关键**。如果稀有度只是换个颜色，用户很快就不在乎了。我们让稀有度影响实际能力：

| 能力 | Common | Epic | Legendary |
|------|--------|------|-----------|
| 称号 | 无 | "敏锐的" | "传奇的" |
| 性格增强 | 单属性 | 双属性 | 全属性 |
| 记忆提取 | 每 3 轮 | 每 2 轮 | 每 2 轮 |
| 视觉特效 | 无 | 紫色光效 | 金色光效 |

Epic 玩家的 Buddy 确实比 Common 更"聪明"（记忆提取频率翻倍），这让稀有度从装饰变成了真实的差异化体验。

## 进化系统：成长曲线的设计

进化门槛的三个维度（记忆数 × 活跃天数 × 对话数）是刻意设计的——每个维度代表不同类型的用户行为：

- **记忆数**：衡量"深度使用"——用户是否真的在用助理管理信息
- **活跃天数**：衡量"持续性"——用户是否养成了每天打开的习惯
- **对话数**：衡量"频率"——用户和 Buddy 的互动密度

三个维度缺一不可。只看天数会奖励"挂机"，只看对话数会奖励"刷量"，只看记忆数会奖励"堆数据"。三个一起看，才能识别出真正的"长期深度用户"。

门槛曲线参考了 Pokémon 的经验值曲线——**早期快速成长，中期稳步推进，后期需要真正的坚持**：

- Common → Uncommon（7 天 + 10 记忆 + 20 对话）：一周入门期，让用户在激动期内看到第一次进化
- Uncommon → Rare（21 天 + 30 记忆 + 50 对话）：三周巩固期，确认用户不是三分钟热度
- Rare → Epic（45 天 + 60 记忆 + 100 对话）：一个半月，进入"日常习惯"阶段
- Epic → Legendary（90 天 + 100 记忆 + 200 对话）：三个月，只有真正的长期用户能达到

**进化不是自动触发的**——用户必须主动点"检查进化"按钮。这是一个刻意的设计：自动进化是一个被动事件（"哦，它变了"），手动检查是一个主动仪式（"让我看看 Buddy 准备好了没有"）。仪式感创造记忆，记忆强化关系。

失败反馈也很重要："记忆 23/30，还差 7 条"比"条件未满足"有用得多——它给用户一个具体的目标和"再坚持一下"的动力。

## 心跳系统：最复杂也最有价值的子系统

心跳是 Buddy 系统中迭代次数最多的部分。从最初的"定时跑个 cron"到最终的"双模式 + 不可见标记 + 服务端数据源 + 互斥守卫"，经历了 5 轮审查才稳定下来。

### 为什么心跳这么难做对

心跳看起来简单——"每天检查一次 HEARTBEAT.md"——但它涉及的状态管理极其复杂：

1. **前后端状态同步**：前端决定"要不要触发"，后端决定"触发后注入什么 prompt"，两边的判断条件必须一致
2. **多触发器互斥**：buddy-welcome 和 heartbeat 共用 autoTrigger flag，同时为 true 时只能走一条路
3. **不可见消息的副作用**：autoTrigger 消息不保存到聊天历史，但 AI 的回复会保存。如果回复包含 HEARTBEAT_OK 或控制标记，这些会泄漏到用户可见的聊天中
4. **完成状态的准确性**：标记"今天做过了"必须基于 AI 确实做了检查，不能因为 AI 在回答中碰巧提到了"记忆"就算完成

### 最终的双模式设计

**完整心跳（autoTrigger）**：空会话 + 有 Buddy + 服务端 `needsHeartbeat = true` → 前端发送不可见消息 `'心跳检查'` → context-assembler 检测到 `userPrompt.includes('心跳检查')` → 注入完整 `buildHeartbeatInstructions()` → AI 自主决定做什么。

这是"Buddy 在等你回来"的场景——用户打开一个空会话，Buddy 已经提前准备好了今天的 briefing。

**软心跳（系统 prompt hint）**：用户主动发消息 + 心跳过期 → prompt 末尾附加一段轻量提示"今天还没做过日常检查，可以顺带看看 HEARTBEAT.md" → AI 在回答用户问题的同时，如果确实做了检查，在回复末尾附加 `<!-- heartbeat-done -->` 不可见标记。

这是"Buddy 在你忙的时候顺手帮你看了一眼"的场景——不打断用户，不抢话题，但确保日常检查不会被遗忘。

### 为什么用 `<!-- heartbeat-done -->` 而不是关键词检测

这个决策经历了三次迭代：

**第一版：无检测**。只要心跳过期的 turn 收到了非空回复，就标记完成。问题：用户问"今天天气怎么样"，AI 回答了天气，心跳就算"完成"了——但 AI 根本没看 HEARTBEAT.md。

**第二版：关键词检测**。检测 `heartbeat|心跳|记忆|memory|检查|提醒` 等词。问题：用户讨论 memory leak、代码检查、日程提醒时全部误判为"心跳完成"。收窄关键词到 `HEARTBEAT.md|日常检查` 又太严——AI 如果用"我看了一下你的待办"这样的自然表达就检测不到。

**第三版：不可见标记**。在 soft hint 中明确告诉 AI：如果你确实做了检查，在回复末尾加 `<!-- heartbeat-done -->`。后端检测这个标记，检测到就更新 `lastHeartbeatDate`。标记在持久化前从所有 contentBlock text 中清除——包括 JSON 序列化路径（tool_use 场景）。

这个方案的精妙之处：**AI 的行为和系统的状态机通过一个约定的信号量解耦**。AI 有自由度决定怎么做检查（可以用自然语言、可以调 MCP 工具、可以简短一提），系统只看"你做了没有"的信号。类似于 HEARTBEAT_OK 在完整心跳中的作用，但更轻量——不需要整个 stripHeartbeatToken 流程。

### `needsHeartbeat` 为什么必须是服务端计算

前端最初自己实现了心跳判定：

```ts
const localToday = `${now.getFullYear()}-${...}`;
const needsHeartbeat = state.heartbeatEnabled && lastHb !== localToday;
```

这导致了两个问题：

1. **UTC 兼容性**：旧版本的 `lastHeartbeatDate` 存的是 UTC 日期字符串，新版本用本地日期。`shouldRunHeartbeat()` 有兼容逻辑同时检查两种格式，前端手写版没有。
2. **activeHours 缺失**：服务端 `shouldRunHeartbeat()` 支持 `activeHours` 配置（只在工作时间触发），前端版不知道这个参数。

改为 `GET /api/settings/workspace` 返回 `needsHeartbeat: !!state.buddy && shouldRunHeartbeat(state)`，前端直接用。**单一数据源比"两边各算一次然后祈祷结果一致"可靠得多。** 这个教训适用于所有"前后端都需要同一个判断结果"的场景。

## 与 Claude Code 宠物系统的对比

Claude Code 在 2025 年末引入了类似的宠物机制，验证了"给 AI 工具加宠物"这个方向的可行性。但两者的设计哲学有根本差异。

### Claude Code 的宠物是一个彩蛋

在终端里，宠物受限于字符渲染——ASCII art 或像素风。它只能在命令执行的输出中短暂出现，终端关了就没了。没有后台存在感、没有主动通知、没有独立的状态面板。

它是一个"用得越多，宠物越大"的奖励机制——本质上是经验条的拟人化。

### CodePilot 的 Buddy 是一个设计支点

桌面客户端的形态让 Buddy 可以拥有 CLI 做不到的存在感：

**空间维度**。Buddy 同时存在于侧栏（始终可见的名字和 emoji）、看板（完整的状态卡片）、聊天（AI 消息旁的头像）、顶栏（一个小 emoji）、系统通知栏（提醒弹窗）。用户不需要"执行一个命令"才能看到 Buddy——它就在那里。

**时间维度**。CLI 宠物只在命令执行期间"活着"。CodePilot 的 Buddy 有心跳系统——即使你不主动对话，Buddy 也在后台检查你的待办、整理记忆、执行定时任务。打开应用时，Buddy 可能已经准备好了今天的 briefing。Electron 的 tray-only 后台模式让 Buddy 即使在无窗口状态下也能弹出系统通知。

**功能维度**。CLI 宠物不影响工具的实际能力。CodePilot 的 Buddy 稀有度直接影响记忆提取频率（Epic+ 每 2 轮 vs 普通每 3 轮），进化解锁增强性格，soul.md 中的 Buddy Trait 直接影响 AI 的回复风格。**Buddy 不是装饰，是影响 AI 行为的参数。**

**视觉维度**。3D Fluent Emoji 在桌面端有完整的视觉表现力——渐变背景、胶囊标签、属性条、进化动画（计划中）。终端的字符渲染无法提供这种视觉丰富度。这不只是审美差异——丰富的视觉让用户更容易对 Buddy 产生情感投射。研究表明，视觉复杂度与情感依附度正相关。

### 本质区别：彩蛋 vs 设计支点

Claude Code 的宠物是在已完成的产品上叠加的趣味层——去掉它，CLI 的功能完全不变。

CodePilot 的 Buddy 是产品架构的一部分——去掉它，记忆、心跳、通知、进化这些系统都需要重新设计它们的用户界面。整个助理产品围绕"有生命感的伙伴"来组织，记忆是 Buddy 的记忆，心跳是 Buddy 的心跳，任务是 Buddy 帮你做的事。

**一个有生命感的伙伴需要一个有空间的家**。终端太小、太临时、太串行，放不下一个值得每天回来看一眼的伙伴。这也是我们选择桌面客户端形态的深层原因——不只是为了"更好的 UI"，而是为了给 Buddy 一个可以"住"的地方。

## 实现后的反思

### 最大的教训：心跳的状态边界比功能逻辑难十倍

整个心跳系统的功能逻辑很简单——"检查 HEARTBEAT.md，有事说事，没事静默"。但围绕它的状态管理经历了 5 轮审查才对齐：

- 第 1 轮：heartbeat 指令混入普通对话（context-assembler 不区分触发类型）
- 第 2 轮：修复后 heartbeat 完全不触发了（autoTrigger 收太紧，没有触发路径）
- 第 3 轮：恢复了触发但 buddy-welcome 和 heartbeat 冲突（同一个 flag 两种语义）
- 第 4 轮：互斥了但前后端判定逻辑分叉（UTC 日期 vs 本地日期）
- 第 5 轮：统一了但软心跳完成标记泄漏到聊天历史（tool_use JSON 路径没清理）

每一轮修复都是"在一个地方堵住了洞，另一个地方漏水"。根因是心跳涉及 4 个文件的 6 个判断点（useAssistantTrigger、context-assembler、route.ts 的触发检测、route.ts 的完成检测、route.ts 的消息保存、workspace API 的 needsHeartbeat），任何一个和其他的不一致都会产生 bug。

**教训**：涉及多文件状态同步的功能，应该在设计阶段画出完整的状态机图——列出所有状态变量、所有转换条件、所有判断点，确认每个判断点看到的状态是一致的。"写代码 → 发现不一致 → 修 → 另一个地方不一致"这种迭代方式成本太高。

### 第二个教训：cron 的边界比想象中远得多

`getNextCronTime()` 从最初的"搜索 48 小时"到最终的"搜索 4 年 + 返回 null"，经历了 4 个版本：

1. 48 小时（2880 分钟暴力扫描）→ 周级 cron 漏过
2. 35 天 → 月级 cron 覆盖了但 `0 9 29 2 *`（闰年 2 月 29 日）漏过
3. 366 天，但分两条路径（wildcard/specific-time），wildcard 路径还是 366 天 → `0 * 29 2 *` 走了 wildcard 还是漏
4. 统一 1461 天单路径 + 日级预检跳过 + 无匹配返回 null

最后一版引入了一个关键的架构决策：**getNextCronTime 返回 `Date | null` 而不是 `Date`**。这迫使所有调用方（`computeNextRun`、`schedule/route.ts`、`notification-mcp.ts`）都必须处理"无有效下次执行时间"的情况——暂停任务或拒绝创建。

之前的 fallback（1h / 30 天 / 4 年）都是同一个错误的变体：**用一个假的执行时间代替"我不知道"**。调度器的执行逻辑是 `next_run <= now` 就触发，任何 fallback 日期最终都会到期。`null` 是唯一不会导致错误执行的返回值。

**教训**：当一个函数不知道答案时，应该明确说"我不知道"（返回 null / 抛异常），而不是编造一个看起来合理的答案。后者会在未来某个不可预测的时间点制造 bug。

### 第三个教训：session-only 任务暴露了"SQLite 是唯一持久化层"的假设

整个任务系统最初假设所有任务都在 SQLite 中。`executeDueTask` 写 SQLite、`computeNextRun` 写 SQLite、`applyBackoff` 写 SQLite、`list` 查 SQLite、`cancel` 删 SQLite。

session-only 任务（`durable=false`）存在内存 Map 中，但调用的是同一个 `executeDueTask`。结果：
- 成功时 SQLite 写入静默失败（task ID 不在表中），但不报错
- 失败时 SQLite 写入也静默失败，错误计数永远是 0，退避永远不生效
- list/cancel 只查 SQLite，session 任务不可见

修复后 `executeDueTask(task, isSessionTask)` 根据 flag 走不同路径——session 任务跳过所有 SQLite 操作，成功/失败都在调用方的内存 Map 中处理。

**教训**：当引入一种新的存储后端（即使只是内存 Map），所有假设"只有一个存储后端"的代码路径都需要审查。不能只在"创建"时处理新后端，"执行/查询/删除/错误处理"每一条路径都需要。

## 更远的思考

### Buddy 是 AI 助理产品的"壳"

往远处想，Buddy 解决的其实不是"留存"问题——它解决的是 AI 助理的**界面范式**问题。

当前所有 AI 产品的界面都是"对话框"——一个输入框、一串消息气泡。这在"问答"场景下是最优的，但在"长期伙伴"场景下是不够的。你不会和一个聊天框建立情感关系，但你可能和一只虚拟猫建立。

Buddy 的真正价值不在于那个 3D emoji 本身，而在于它提供了一个**拟人化的界面层**——记忆变成了"它记得你"，心跳变成了"它在等你"，进化变成了"它在成长"，通知变成了"它在叫你"。同样的功能，换一种叙事方式，用户体验完全不同。

如果这个方向是对的，那 Buddy 系统未来的发展方向不是"加更多游戏化元素"，而是**让 Buddy 成为 AI 所有行为的统一叙事者**。AI 在后台做了什么、学到了什么、发现了什么——都通过 Buddy 的视角传达给用户。Buddy 不是 AI 的装饰，是 AI 的脸。

### "有生命感"的具体含义

什么让一个虚拟实体有"生命感"？从 Tamagotchi 到 Neopets 到 Pokémon，有几个共性：

1. **状态随时间变化**——不被照顾就退化。目前 Buddy 缺少这个（永远健康）
2. **行为有不可预测性**——偶尔做出"意外"的事。目前 AI 天然具备这个
3. **对用户的行为有回应**——忽略它会生气，关心它会高兴。目前只有心跳这一个回应链路
4. **有自己的节奏**——不是"你按按钮它才动"。心跳和定时任务部分实现了这个
5. **成长可见**——从小变大、从弱变强。进化系统覆盖了这个

目前 Buddy 在 2、4、5 上做得不错，1 和 3 还很薄弱。未来可以考虑：
- **忽略退化**：连续 N 天不打开 → Buddy 属性缓慢下降 → 回来后有"它想你了"的表达
- **互动回应**：摸头、喂食等小互动 → Buddy 有即时的视觉反馈 → 属性微量恢复
- **情绪状态**：基于最近的互动频率、任务完成度、记忆增长，Buddy 有不同的"情绪"，影响回复语气

### 从个人助理到 AI 原生桌面

CodePilot 目前有：项目聊天、个人助理、看板、CLI 工具管理、远程桥接、Skills、媒体管线、生成式 UI。这些功能的共同点是它们都围绕"AI 作为操作者"来设计——用户说意图，AI 执行。

如果把 Buddy 的设计思路（拟人化界面层）推广到整个产品，CodePilot 可能演变成一种新的桌面范式：**AI 原生桌面**。

传统桌面 OS 的范式是"窗口 + 图标 + 菜单"——用户通过图形界面操控应用。AI 原生桌面的范式可能是"伙伴 + 对话 + 看板"——用户通过自然语言和 AI 伙伴协作，AI 通过看板和生成式 UI 展示它的理解和成果。

这不是一个短期计划，但它是 Buddy 设计方向的自然延伸：**如果 AI 的行为需要一个拟人化的界面层，那整个桌面的交互方式都应该围绕这个界面层重新设计。**

### 信任与透明度的边界

Buddy 的拟人化有一个风险：用户可能过度信任它。"我的猫猫说记住了"比"AI 已保存到 memory.md"更容易被接受，但也更容易掩盖问题——如果记忆保存失败了呢？如果心跳检查遗漏了重要事项呢？

长远来看，Buddy 的设计需要在"有生命感"和"透明可控"之间找平衡：
- 功能成功时用 Buddy 的语言传达（"我记住了！"）
- 功能失败时用系统的语言传达（"记忆保存失败：磁盘已满"）
- 重要操作前用系统的语言确认（"即将删除 23 条旧记忆，确认？"）

**Buddy 是 AI 的脸，但当出了问题时，用户需要看到脸后面的机器。** 这个设计原则会指导未来所有 Buddy 相关功能的错误处理方式。
