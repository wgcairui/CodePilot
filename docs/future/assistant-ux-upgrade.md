# 助理体验升级：视觉人格化 + Onboarding Wizard + 引导策略

> 目标：让助理从"侧栏里的一个灰色项目"变成"有名字、有头像、有存在感的个人 AI"

## 一、视觉与人格化

### 1.1 助理头像

**方案：Identicon 确定性生成**

参考 Vercel identicon prototype（https://identicon-prototype.labs.vercel.dev/）：
- 用助理名字 hash → 确定性生成头像（同名字同头像）
- OKLCH 色彩空间（与 CodePilot 设计系统一致）
- 纯前端 Canvas 渲染，不需要外部服务

可选的开源库：
- `@dicebear/core` + `@dicebear/collection` — 多种风格（像素、抽象、Bottts 等），支持 SVG
- `jdenticon` — 几何 identicon，轻量
- 自建：Murmur3 hash → 颜色 + 简单几何图案（像 Vercel 的 Pixel Sprite 算法）

头像显示位置：
- 侧栏助理项目（替代现在的文件夹图标）
- 聊天消息（AI 回复旁）
- 设置页

### 1.2 侧栏助理项目重设计

**现在：**
```
📁 /path/to/workspace  [助理]    ← 灰色，和普通项目一样，右侧小字"助理"
```

**改为：**
```
┌─────────────────────────────────┐
│ 🤖 小助                         │  ← 头像 + 助理名字（或"个人助理"）
│ ○ 上次心跳：2h前 · 记忆 23 条    │  ← 状态行
│   /path/to/workspace            │  ← 路径（小字灰色）
└─────────────────────────────────┘
```

**关键设计决策：**

| 维度 | 决策 |
|------|------|
| 名字来源 | soul.md 里的助理名字 → 未设置时显示"个人助理" |
| 主题色 | 助理项目用 `--primary` 色调高亮，区别于普通项目的灰色 |
| 聊天主题色 | 助理聊天窗口的 header/边框带主题色 |
| 右侧标记 | 去掉"助理"文字标记，用头像 + 主题色识别 |
| 置顶 | 助理项目始终置顶在项目列表最上方 |

**状态行内容（两行小字）：**
- 第一行：最近活动——"上次心跳：2h 前" / "今天聊过 3 轮" / "未设置 → 点击开始"
- 第二行：记忆统计——"记忆 23 条" / "待办 2 项"
- 如果太慢可简化为一行 + 状态点（绿=活跃 / 灰=未配置）

### 1.3 聊天窗口同步

助理项目的聊天窗口视觉差异化：
- AI 消息旁显示助理头像（普通项目用默认图标）
- 顶部 bar 带主题色底边
- 空状态显示助理头像 + 名字（而非通用的 CodePilot logo）

已有项目但未配置助理的用户：

## 四、Onboarding Wizard（前端组件，非 AI 对话）

### 4.1 设计理念

**放弃 AI 对话式 onboarding**，改用前端 Wizard 组件：
- 每个选择实时落库，不等全部答完
- 零 API 调用，秒完成
- 像填一个可爱的小表单，不像在和 AI 聊天
- 用户感知：3 步完成，每步 2 秒

### 4.3 文件生成策略

| 文件 | 何时生成 | 需要 AI？ |
|------|---------|----------|
| user.md | Step 1 完成后 | ❌ 模板填充 |
| soul.md | Step 2 完成后 | ❌ 模板填充 |
| claude.md | Step 3 确认后 | ❌ 系统预设模板（已有 5 个区块） |
| memory.md | Step 3 确认后 | ❌ 空模板 + 初始条目 |
| HEARTBEAT.md | Step 3 确认后 | ❌ 默认模板 |
| config.json | Step 3 确认后 | ❌ 默认配置 |

**零 API 调用**。全部用模板填充。深度个性化交给渐进式更新。

### 4.4 与现有 AI 对话式 onboarding 的关系

**Wizard 替代 AI 对话式**——不是共存。原因：
- AI 对话式依赖 API 调用（慢、要钱、可能失败）
- AI 控制不了问题数量（Codex 审核发现问了 8 个问题）
- 用户不知道什么时候结束
- 新用户可能不知道怎么和 AI 聊

Wizard 完成后，用户进入聊天，AI 通过渐进式更新指引自动学习更深层的偏好。

### 4.5 现有 context-assembler onboarding prompt 的处理

Wizard 完成后 `state.onboardingComplete = true`，AI 对话式 onboarding 不再触发。`buildOnboardingInstructions()` 保留作为 fallback（如果用户通过 API 或旧版本创建 workspace）。

## 五、记忆驱动的 Quick Actions

### 5.1 输入框上方的预设问题

助理聊天的空输入状态下，显示 2-3 个 quick action chip：

```
┌─────────────────────────────────────┐
│  💡 "周五开会的材料准备好了吗？"      │
│  💡 "Rust 学习进展"                  │
│  💡 "回顾本周"                       │
└─────────────────────────────────────┘
│ 输入消息...                          │
```

### 5.2 生成策略（纯前端，不调 API）

```typescript
function generateQuickActions(workspacePath: string): string[] {
  const actions: string[] = [];

  // 1. 从 daily memory 提取未完成事项
  const dailyFiles = loadDailyMemories(workspacePath, 3);
  for (const daily of dailyFiles) {
    const todos = daily.content.match(/- \[ \] (.+)/g);
    if (todos) {
      actions.push(todos[0].replace('- [ ] ', ''));
    }
  }

  // 2. 从 user.md 提取 goals
  const userMd = fs.readFileSync(path.join(workspacePath, 'user.md'), 'utf-8');
  const goalMatch = userMd.match(/## (?:当前目标|Current Goals)\n([\s\S]*?)(?=\n##|$)/);
  if (goalMatch) {
    const firstGoal = goalMatch[1].trim().split('\n')[0].replace(/^[-*] /, '');
    if (firstGoal) actions.push(firstGoal);
  }

  // 3. 固定的 quick action
  actions.push('回顾本周');

  return actions.slice(0, 3);
}
```

纯文件读取，不调 API，<50ms 完成。

### 5.3 API 端点

```
GET /api/workspace/quick-actions
→ { actions: ["周五开会的材料准备好了吗？", "Rust 学习进展", "回顾本周"] }
```

前端在助理聊天页 mount 时调用，chip 渲染在 MessageInput 上方。

## 七、实施顺序

```
Phase A: Onboarding Wizard（核心，替代 AI 对话式）
  → 前端组件 + 模板文件生成 + state 管理

Phase B: 侧栏视觉重设计
  → Identicon + 名字 + 状态行 + 主题色 + 置顶

Phase C: 聊天主题色 + Quick Actions
  → 消息头像 + 顶部 bar 主题色 + 记忆驱动 chip

Phase D: 用户引导
  → 新用户空状态 + 老用户侧栏卡片

Phase E: 助理状态面板
  → 右侧面板替换 + 状态/记忆/设置 tab
```

Phase A 最重要——它直接解决"onboarding 太重、问题太多、结束不明确"的问题。
Phase B 其次——它解决"用户发现不了助理"的问题。
其余按优先级排。
