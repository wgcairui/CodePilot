## 一、用户发现的问题

### U1. 看板面板生成式 UI 样式丢失 [P1]

**现象：** 看板（Dashboard）面板里的 Widget 只显示纯文本，没有样式渲染。

**可能原因：**
- `WidgetRenderer.tsx` 在本轮有改动（line 2 变更），可能影响了 iframe 渲染
- 或者 `widget-sanitizer.ts` 的改动（line 12 变更）过度清理了样式
- 需要用 CDP 截图确认具体渲染状态

**影响范围：** 所有使用看板面板的用户

### U2. 部分用户 OpenAI 授权登录 403 [P1]

**现象：** 部分用户在 OAuth 登录时收到 403 错误，但开发环境正常。

**可能原因：**
- OpenAI OAuth 的 `CLIENT_ID = 'app_EMoamEEZ73f0CkXaXp7hrann'` 是 Codex CLI 的公开 Client ID，可能有地域/账号类型限制
- ChatGPT Free 用户可能无权使用 Codex API（仅 Plus/Pro）
- callback server 绑定 `127.0.0.1:1455`，某些环境可能无法访问
- token exchange 可能在特定网络环境下被拦截

**需确认：** 报 403 的用户是 Plus 还是 Free 账号；是 OAuth 阶段还是 API 调用阶段报错

### U3. 服务商连接测试误报失败 [P2]

**现象：** 已保存的服务商配置，运行连接测试报错，但实际能正常发消息。

**可能原因：**
- 连接测试和实际聊天走的 provider resolution 路径不同
- 测试可能用的是旧的 `toClaudeCodeEnv()` 路径而聊天走的新 `toAiSdkConfig()` 路径
- 或者测试请求没带正确的 auth headers

**关联 Issue：** #437（0.48.0 前就有报告）

## 二、GitHub Issues（0.48.0 后新增）

### #456 [P0] 双引擎切换导致第三方 Provider 认证死循环

**报告者描述：**
1. 新版强制要求 Provider 实体存在，否则拦截 CLI 请求
2. 设置第三方 Provider 后，AI SDK 引擎错误地将请求封装为 Anthropic 格式
3. CLI 引擎找不到 active provider 就不注入环境变量

**分析：** 这是本轮重构的核心回归。`resolveRuntime` 的 auto 模式 + `predictNativeRuntime` + MCP 加载的三层判断逻辑在第三方 Provider 下可能没有正确路由。特别是：
- 用户只有第三方 Provider（如智谱），没有 Anthropic 凭据
- auto 模式检测到 CLI 可用 → 选 SDK runtime
- 但 SDK runtime 需要 Anthropic 凭据才能跑
- 结果：两个引擎都不能用

**根因假设：** auto 模式应该在没有 Anthropic 凭据时 fallback 到 native runtime，当前逻辑只检查 SDK 是否 available（CLI 二进制存在），不检查凭据。

### #455 [P2] 多 Bridge 适配器同时启用导致互相干扰

**报告者描述：** 飞书 + QQ 同时启用 → 两个都挂。禁用一个后另一个也不恢复，需重启。

**分析：** Bridge adapter 的生命周期不隔离。一个 adapter 的重连循环影响了 Bridge Manager 的消息路由。这不是 0.48.0 引入的问题，但用户在这个版本报告。

### #454 [P2] Windows 11 卸载卡住

**分析：** 和本轮改动无关，NSIS 安装器的已知问题。

### #453 [P3] 不支持 Claude Code 的 /btw 指令

**分析：** Claude Code 的 slash command 在 SDK runtime 下工作，native runtime 下未实现。这是预期的能力差异，需要在 Skill 系统中补齐或明确标注。

## 三、Sentry 线上错误监控

### S1. [107x] AI_NoOutputGeneratedError [P0] 🆕

```
AI_NoOutputGeneratedError: No output generated. Check the stream for errors.
首次出现：2026-04-09（发布当天）
```

**分析：** 这是 native runtime agent-loop 的 `EMPTY_RESPONSE` 错误。107 次出现说明有大量用户的 native runtime 调用没有返回内容。可能原因：
- 第三方代理不支持某些模型 ID
- provider 配置协议不匹配（如 sdkProxyOnly 的 provider 被 native runtime 调用）
- Codex API 的模型 ID 不被识别

### S2. [45x] No provider credentials available [P1] 🆕

```
Error: No provider credentials available. Please configure a provider in Settings or set ANTHROPIC_API_KEY.
首次出现：2026-04-09
```

**分析：** 直接关联 #456。native runtime 的 `createModel()` 在找不到凭据时抛出此错误。说明有用户的 provider 配置在 native 路径下没被正确解析。

### S3. [6x+] AI_MissingToolResultsError [P1] 🆕

```
AI_MissingToolResultsError: Tool result is missing for tool call call_xxx
首次出现：2026-04-09
```

**分析：** AI SDK streamText 的 tool call 返回了结果但 SDK 没收到。可能是：
- 权限审批超时后 tool result 丢失
- 并发 tool call 下某个 result 没被正确返回
- 也可能是第三方代理返回了不完整的 tool call 响应

### S4. [28x] Controller is already closed [P1]

```
TypeError: Invalid state: Controller is already closed
首次出现：2026-04-05（早于 0.48.0）
```

**分析：** ReadableStream controller 在流结束后仍有写入尝试。agent-loop 的 keep_alive timer 或 onStepFinish callback 可能在 stream close 后触发。0.48.0 前已存在但可能加剧。

### S5. [8x] Model not found: sonnet [P1]

```
HTTP 404: Not found the model sonnet or Permission denied
HTTP 502: Model not found(sonnet). 无可用模型 sonnet
```

**分析：** 短别名 `sonnet` 没有被解析为完整模型 ID `claude-sonnet-4-5-20250929`。native runtime 的 `createModel()` 在 `isShortAlias()` 中有映射，但某些路径可能绕过了。或者第三方代理不接受短别名。

### S6. [4x] HMAC signature apikey not found [P2]

```
HTTP 401: HMAC signature cannot be verified: apikey not found
```

**分析：** 某个 Provider（可能是火山引擎）的认证方式不匹配。native runtime 可能用了错误的 auth 头。

### S7. [6640x] Claude Code process exited with code 1 [既有]

```
Error: Claude Code process exited with code 1
首次出现：2026-04-04（早于 0.48.0）
```

**分析：** SDK runtime 既有问题，非本轮引入。但量级（6640 次）说明这是用户最常见的错误。

### S8. [1x] Claude Code compat API 500 [P2] 🆕

```
Error: Claude Code compat API error: 500 Internal Server Error
```

**分析：** ClaudeCodeCompatAdapter（专门适配第三方代理的 adapter）返回 500。可能是代理服务器端问题，也可能是我们的请求格式不正确。

## P0 问题初步对策

### #456 + 45x 凭据丢失

根因：`resolveRuntime()` 的 auto 模式只检查 CLI 二进制是否存在，不检查 Anthropic 凭据。用户只有第三方 Provider 时，auto 选了 SDK 但 SDK 没有 Anthropic 凭据 → 失败。

**对策方向：**
- auto 模式应该增加凭据检查：有 Anthropic 凭据 → SDK；否则 → native
- 或者：SDK runtime `isAvailable()` 应该检查凭据不仅是 CLI 二进制

### 107x AI_NoOutputGeneratedError

根因可能多样：
- sdkProxyOnly provider 被 native runtime 调用（应该走 ClaudeCodeCompatAdapter 但没有）
- 模型 ID 不被第三方代理识别
- 请求格式不匹配

**对策方向：**
- 在 Sentry 上报中加 provider/model 信息，定位具体是哪些 provider 出的问题
- 检查 sdkProxyOnly provider 在 native 路径下的处理逻辑
