## 一、用户发现的问题

### U1. 看板面板生成式 UI 样式丢失 [P1]

**现象：** 看板（Dashboard）面板里的 Widget 只显示纯文本，没有样式渲染。

**可能原因：**
- `WidgetRenderer.tsx` 在本轮有改动（line 2 变更），可能影响了 iframe 渲染
- 或者 `widget-sanitizer.ts` 的改动（line 12 变更）过度清理了样式
- 需要用 CDP 截图确认具体渲染状态

**影响范围：** 所有使用看板面板的用户

### U2. 部分用户 OpenAI 授权登录 403 [P1]

**现象：** 部分用户在 OAuth 登录时收到 403 错误，但开发环境正常。

**可能原因：**
- OpenAI OAuth 的 `CLIENT_ID = 'app_EMoamEEZ73f0CkXaXp7hrann'` 是 Codex CLI 的公开 Client ID，可能有地域/账号类型限制
- ChatGPT Free 用户可能无权使用 Codex API（仅 Plus/Pro）
- callback server 绑定 `127.0.0.1:1455`，某些环境可能无法访问
- token exchange 可能在特定网络环境下被拦截

**需确认：** 报 403 的用户是 Plus 还是 Free 账号；是 OAuth 阶段还是 API 调用阶段报错

### U3. 服务商连接测试误报失败 [P2]

**现象：** 已保存的服务商配置，运行连接测试报错，但实际能正常发消息。

**可能原因：**
- 连接测试和实际聊天走的 provider resolution 路径不同
- 测试可能用的是旧的 `toClaudeCodeEnv()` 路径而聊天走的新 `toAiSdkConfig()` 路径
- 或者测试请求没带正确的 auth headers

**关联 Issue：** #437（0.48.0 前就有报告）

## 二、GitHub Issues（0.48.0 后新增）

### #456 [P0] 双引擎切换导致第三方 Provider 认证死循环

**报告者描述：**
1. 新版强制要求 Provider 实体存在，否则拦截 CLI 请求
2. 设置第三方 Provider 后，AI SDK 引擎错误地将请求封装为 Anthropic 格式
3. CLI 引擎找不到 active provider 就不注入环境变量

**分析：** 这是本轮重构的核心回归。`resolveRuntime` 的 auto 模式 + `predictNativeRuntime` + MCP 加载的三层判断逻辑在第三方 Provider 下可能没有正确路由。特别是：
- 用户只有第三方 Provider（如智谱），没有 Anthropic 凭据
- auto 模式检测到 CLI 可用 → 选 SDK runtime
- 但 SDK runtime 需要 Anthropic 凭据才能跑
- 结果：两个引擎都不能用

**根因假设：** auto 模式应该在没有 Anthropic 凭据时 fallback 到 native runtime，当前逻辑只检查 SDK 是否 available（CLI 二进制存在），不检查凭据。

### #455 [P2] 多 Bridge 适配器同时启用导致互相干扰

**报告者描述：** 飞书 + QQ 同时启用 → 两个都挂。禁用一个后另一个也不恢复，需重启。

**分析：** Bridge adapter 的生命周期不隔离。一个 adapter 的重连循环影响了 Bridge Manager 的消息路由。这不是 0.48.0 引入的问题，但用户在这个版本报告。

### #454 [P2] Windows 11 卸载卡住

**分析：** 和本轮改动无关，NSIS 安装器的已知问题。

### #453 [P3] 不支持 Claude Code 的 /btw 指令

**分析：** Claude Code 的 slash command 在 SDK runtime 下工作，native runtime 下未实现。这是预期的能力差异，需要在 Skill 系统中补齐或明确标注。


## P0 问题初步对策

### #456 + 45x 凭据丢失

根因：`resolveRuntime()` 的 auto 模式只检查 CLI 二进制是否存在，不检查 Anthropic 凭据。用户只有第三方 Provider 时，auto 选了 SDK 但 SDK 没有 Anthropic 凭据 → 失败。

**对策方向：**
- auto 模式应该增加凭据检查：有 Anthropic 凭据 → SDK；否则 → native
- 或者：SDK runtime `isAvailable()` 应该检查凭据不仅是 CLI 二进制

### 107x AI_NoOutputGeneratedError

根因可能多样：
- sdkProxyOnly provider 被 native runtime 调用（应该走 ClaudeCodeCompatAdapter 但没有）
- 模型 ID 不被第三方代理识别
- 请求格式不匹配

**对策方向：**
- 在 Sentry 上报中加 provider/model 信息，定位具体是哪些 provider 出的问题
- 检查 sdkProxyOnly provider 在 native 路径下的处理逻辑

### 5.2 resolver 语义 bug（已确认）+ 38x Sentry bucket 归因（待确认）

> Codex review 修正（2026-04-11 下午）：最初这一节把 "resolver `is_active` bug" 写成 "38x `No provider credentials available` 的共同根因"，这个归因太过头。以下把两件事拆开。

#### 5.2.1 已确认：resolver 的 `is_active` 语义 bug

用户 #461 的 doctor JSON 反复打日志：
```
[provider-resolver] Default provider "Aliyun Bailian" (c1ed08a2***) is inactive, falling back
```

追 `src/lib/provider-resolver.ts:107-142` 发现两套**独立不同步**的语义：

| 字段 | 语义 | 设置入口 |
|---|---|---|
| `default_provider_id` (setting 表) | 全局默认 provider | `/api/providers/set-default`, Settings dropdown |
| `api_providers.is_active` (列) | radio-button 风格"当前激活"（一次只一个 = 1） | `/api/providers/[id]/activate` |

**关键代码证据：**

- `src/lib/db.ts:1358` — `createProvider()` 插入时 `is_active` 硬编码为 `0`
- `src/lib/db.ts:1514-1525` — `activateProvider()` 事务 `UPDATE api_providers SET is_active = 0; UPDATE ... WHERE id = ? SET is_active = 1`
- `src/lib/provider-resolver.ts:132-136` — `default_provider_id` 分支检查 `defaultProvider?.is_active`，为 false 则 warn 并 fallback

**语义冲突路径：**
1. 创建百炼 provider → `is_active = 0`
2. 在 Settings 里选百炼为默认 → 调 `setDefaultProviderId()`，**不触发 `activateProvider`**
3. 数据库状态：`default_provider_id = bailian_id`，`bailian.is_active = 0`
4. 新对话 + 客户端未显式带 `provider_id` → `resolveProvider()` 走 line 126 `else if (!effectiveProviderId)` 分支 → 看 `is_active=0` → 忽略默认值 fallback

**这个 bug 是实锤**：两套字段的语义冲突、用户明确设置被 resolver 忽略、doctor JSON 的日志都坐实。

#### 5.2.2 不成立：这不是 Sentry `No provider credentials available` 38x bucket 的**直接**链路

最初文档声称"`claude-client.ts` → `resolveForClaudeCode()` → `resolveProvider()` 再走一遍 `is_active` 过滤导致 SDK 注入失败"。核对代码后**这条链路讲不通**：

| 论点 | 代码证据 |
|---|---|
| 精确错误字符串只在一处抛 | `src/lib/ai-provider.ts:64-68` — `if (!resolved.hasCredentials && !resolved.provider) throw new Error('No provider credentials available...')`。这是 **native runtime 的 `createModel()`**，不是 SDK runtime |
| chat 主流程传的是显式 providerId | `src/app/api/chat/route.ts:169` `const effectiveProviderId = provider_id || session.provider_id \|\| ''`<br>`src/app/api/chat/route.ts:170-175` `resolveProviderUnified({ providerId: effectiveProviderId \|\| undefined, ... })` |
| resolver 对显式 providerId 本来就跳过 inactive 检查 | `src/lib/provider-resolver.ts:95` `const isExplicitRequest = !!opts.providerId;`<br>`src/lib/provider-resolver.ts:109` `if (provider && !provider.is_active && !isExplicitRequest)` — **只有非显式请求才过滤** |

**结论：** 只要 session 有 `provider_id` 或 client 发了 `provider_id`，`isExplicitRequest=true`，resolver 的 `is_active` 过滤**不会触发**。真正会走到 `!effectiveProviderId` default 分支并被 `is_active=0` 干掉的窗口是：

- 新对话（无 session）
- 客户端没带 `provider_id`（依赖全局 default）
- `default_provider_id` 指向一个 `is_active=0` 的 provider

这个场景**存在**，但它只能解释一部分新对话首条消息失败，解释不了 bucket 总量 143x（含 0.48.0 105x + 0.48.1 38x）。而且 bucket 的精确错误字符串 `No provider credentials available` 只在 native runtime 的 `ai-provider.ts:64` 抛出 — 需要进一步定位是什么场景让 `createModel()` 拿到了一个空的 `ResolvedProvider`。

#### 5.2.3 行动项

1. **`src/lib/provider-resolver.ts:132-136` 的 `is_active` 过滤确实要修**，作为 5.2.1 的 P0 修复。效果：`default_provider_id + is_active=0` 的用户从此能正常发送新对话首条消息。
2. **bucket 归因暂缓定论**，改为"高概率子根因之一"。在 Sentry 上报里补以下 tag：
   - `runtime.name`（sdk / native）
   - `provider.id`, `provider.protocol`, `provider.is_active`
   - `resolve.path`（explicit / session / default / env）
   - `default_provider_id`, `sessionProviderId`
3. **拆两条独立任务：** (a) resolver 语义修复 (b) native bucket `No provider credentials` 精确归因。两者可能有交集但不是同一个。

### 5.4 #448 根因：关闭窗口后丢失 last session URL

> 用户报告：mac 上点 ❌ 关闭 → dock 再点开 → 停在 "starting codepilot" loading → 加载完后回到新对话页而不是上次的 session。

**完整链路（已确认每一步都是代码现状，非推测）：**

**关闭流程** (`electron/main.ts:727, 1437-1452`)
1. 红色 ❌ → `window-all-closed` 事件触发
2. bridge 非 active → `killServer()` 同步等 3 秒 → Next.js utility process SIGTERM
3. `process.platform === 'darwin'` → **不 `app.quit()`**，app 留在 dock
4. `mainWindow.on('closed')` → `mainWindow = null`，`serverProcess = null`

**重开流程** (`electron/main.ts:1455-1479`)
1. 点 dock 图标 → `app.on('activate')`
2. `BrowserWindow.getAllWindows().length === 0` → 重启分支
3. `startServer(port)` + `createWindow()` **无参数** → `mainWindow.loadURL(LOADING_HTML)` 显示 "starting codepilot" splash
4. `waitForServer(port, 30000)` 每 300ms 轮询 `/api/health`（冷启动 Next.js 本身需要几秒 — 这就是用户感觉"每次都很久"的原因）
5. server 就绪 → `mainWindow.loadURL('http://127.0.0.1:${port}')` — **加载 root URL，不带任何 path**
6. root URL → `src/app/page.tsx:1-5` → `redirect('/chat')`
7. `/chat` 是新对话页 (`src/app/chat/page.tsx`)，**不加载任何历史 session**

**为什么 session 没恢复：**

`grep -r 'last-session\|lastSessionId'` 结果：**0 处持久化 last session id**。前端 localStorage 存了：
- `codepilot:last-model`
- `codepilot:last-provider-id`
- `codepilot:last-working-directory`
- `codepilot:last-model-provider`

**唯独没存 `last-session-id` 或 `last-url`**。所以每次冷启动都必然回到 `/chat` 新对话页。

**为什么用户感觉"默认选中助手"：** `/chat` 新对话页在空状态下显示 `ChatEmptyState` 或 `OnboardingWizard`，且 UnifiedTopBar 会显示 "Assistant"/"Dashboard" tab。用户看到的不是字面的 assistant workspace，而是空白的 `/chat` 首屏被他解读为"助手"。

**为什么跨版本都有：** `electron/main.ts` 最近一次改动是 `7145760 fix: Sentry opt-out global enforcement`（v0.46 前），再往前是 v0.44 的 Sentry 接入。**v0.46/0.47/0.48 整个窗口 + server 生命周期代码都没动过**，这和用户"0.46/0.47/0.48 都一样"的反馈完全吻合。

**修复方案：**

> Codex review 修正（2026-04-11）：最初写的 Fix A 是 "`chat/page.tsx` mount 时无条件 `router.replace` 到 last session id"。这会劫持所有合法的"新对话"入口 —— 见下方"误伤清单"。Fix B 写的是 `did-navigate`，但前端 in-page 路由（settings/bridge）都是 `window.history.replaceState`，不触发这个事件。以下按 Codex 建议改成**冷启动恢复标记 + 组合 in-page 事件**。

**Fix A（仅冷启动恢复）**

实施步骤：

1. **Electron 冷启动**：`app.on('activate')` 里 loadURL 时带查询参数 `?resume=1`，表明这是"从关闭态恢复"，不是用户主动导航
   ```ts
   mainWindow.loadURL(`http://127.0.0.1:${port}/?resume=1`);
   ```
2. **`src/app/chat/[id]/page.tsx`** mount 时 `localStorage.setItem('codepilot:last-session-id', id)`
3. **`src/app/chat/page.tsx`** mount 时**只在 `resume=1 且无 prefill` 时**才尝试恢复：
   ```ts
   const params = new URLSearchParams(window.location.search);
   const isResume = params.get('resume') === '1';
   const hasPrefill = params.has('prefill');
   if (isResume && !hasPrefill) {
     const lastId = localStorage.getItem('codepilot:last-session-id');
     if (lastId) {
       // 先验证 session 存在
       fetch(`/api/chat/sessions/${lastId}`, { method: 'HEAD' }).then(r => {
         if (r.ok) router.replace(`/chat/${lastId}`);
       });
     }
   }
   ```
4. 恢复成功后**清掉 `?resume=1`**（避免页面内部 navigation 误触发）

**为什么必须 gate 在 `resume=1` 和 `!prefill`：** `/chat` 是多个合法新对话入口的目标，按无条件 router.replace 会误伤：

| 入口 | 代码位置 | 期望行为 |
|---|---|---|
| CLI Tools 发送调试命令 | `src/components/cli-tools/CliToolsManager.tsx:143, 151` | `window.location.href = '/chat?prefill=...'` — 要新对话 |
| CLI Tool Detail 对话框 | `src/components/cli-tools/CliToolDetailDialog.tsx:227` | 同上 |
| CLI Tool Extra Detail | `src/components/cli-tools/CliToolExtraDetailDialog.tsx:49` | 同上 |
| 错误页"Start a new chat"链接 | `src/app/chat/[id]/page.tsx:164` | `<Link href="/chat">` — 要新对话 |
| 用户主动点侧栏"新对话"按钮 | Various | 要新对话 |

无 `resume=1` 标记的情况 = 用户或代码主动导航到 `/chat` = 必须保持新对话语义，**不能**被 last session 劫持。

**Fix B（Electron 覆盖全路径 — 需要监听 in-page 事件）**

实施步骤：

1. **renderer 上报 pathname**：最稳的做法是 `src/app/layout.tsx` 或一个全局 hook 监听 `pathname` + `window.location.hash` 变化，通过 IPC 上报给 main
   ```ts
   // renderer 侧
   useEffect(() => {
     window.electronAPI?.reportRoute?.(window.location.pathname + window.location.hash);
   }, [pathname]);
   ```
2. **Electron main 接收并存盘** 到 `userData/last-route.txt`
3. **`app.on('activate')`** 读取 last-route，拼成 `http://127.0.0.1:${port}${lastRoute}?resume=1`

**为什么不能只用 `did-navigate`：** 当前代码里 settings 和 bridge 的子页切换用的是 in-page 路由：

- `src/components/settings/SettingsLayout.tsx:66` — `window.history.replaceState(null, "", '/settings#${section}')`
- `src/components/bridge/BridgeLayout.tsx:65` — `window.history.replaceState(null, "", '/bridge#${section}')`

`history.replaceState` **不触发** `did-navigate`。如果必须用 webContents 事件，最低要求是**同时监听** `did-navigate` + `did-navigate-in-page`。但更稳的是 renderer 主动上报（对 `replaceState` 有效，对 hash 变化有效，对未来新增的 SPA 路由也有效）。

`electron/main.ts:712` 目前只有一个 `will-navigate` 监听（用于拦截外链跳转到系统浏览器），没有任何路由记录机制。

**Fix C（可选增益）"保持后台运行" 设置**

关窗时不 kill server，保留后台。彻底消除 "starting codepilot" 等待时间（几秒的 Next.js 冷启动）。需要 tray 管理（类似现在 bridge 后台模式）。作为独立的 v0.49+ feature，不塞进 v0.48.2 hotfix。

**推荐路径：v0.48.2 只做 Fix A + 前端挂 last-session 写入；Fix B 延后到 v0.49。** Fix A 解 80%+ 场景（chat session 恢复），Fix B 的 settings deep link 恢复是长尾。

### 5.5 #449 根因：`/api/providers/test` 用 mask 值作为真 API key

> 用户报告：创建第三方 provider → 测试成功 → 保存 → 编辑 → 测试失败 → 重新输入 key → 成功。以为 API key 没持久化。

**追查结果：持久化链路完全正确。** 真正的 bug 在连接测试端点。

**证据链：**

1. **PUT route 防 mask** (`src/app/api/providers/[id]/route.ts:52-55`)：
   ```ts
   if (body.api_key && body.api_key.startsWith('***')) {
     delete body.api_key;
   }
   ```
2. **`updateProvider` 防覆盖** (`src/lib/db.ts:1383`)：`const apiKey = data.api_key ?? existing.api_key`

**但 `/api/providers/test` 没防** (`src/app/api/providers/test/route.ts:15-26`)：
```ts
const { presetKey, apiKey, baseUrl, protocol, ... } = body;
// ...
const result = await testProviderConnection({
  apiKey: apiKey || '',  // ← 直接用 body 里的 key，不查 DB
  ...
});
```

**实际 bug 路径：**
1. 创建 provider，UI 输入 `sk-ant-xxx-abcdef` → POST `/api/providers/test` 带真 key → ✅ 成功
2. 保存：PUT 正确写入 DB（mask 防护生效）
3. 点编辑：GET `/api/providers/[id]` 经 `maskApiKey()` 返回 `***xcdefgh`
4. UI 把 `***xcdefgh` 填进 api_key 输入框
5. 点"测试连接"：POST `/api/providers/test`，body 里 `apiKey = '***xcdefgh'`
6. test route 把 mask 值当真 key 直接打第三方 API → **401 Authentication failed**
7. 用户以为"没持久化"，重新输入真 key → 成功

DB 里的数据一直是对的，只是测试那一次用错了 key。

**修复方案（A 后端 + B 前端必须同时做）：**

> Codex review 修正（2026-04-11）：最初版本只说"后端接受 providerId、前端不发 apiKey"。但后端 `test/route.ts:17` 的 `!apiKey` 拦截写在最前面，前端若只是简单"不发 apiKey" 会立刻被 `NO_CREDENTIALS` 挡下；另外前端编辑态把 mask 值直接塞进 state（`PresetConnectDialog.tsx:154`、`ProviderForm.tsx:98`），保存/测试都原样发出，只改 placeholder 修不到根上。以下把细节补齐。

**Fix A — 后端 `src/app/api/providers/test/route.ts` 改造：**

1. 增加可选 body 字段 `providerId`
2. **调整 `!apiKey` 拦截的时机** — 必须发生在 DB 回填**之后**：
   ```ts
   const { providerId, presetKey, apiKey: bodyApiKey, baseUrl, ... } = body;

   // Step 1: 先尝试从 DB 回填（若 apiKey 缺失或是 mask）
   let effectiveApiKey = bodyApiKey;
   if (providerId && (!effectiveApiKey || effectiveApiKey.startsWith('***'))) {
     const stored = getProvider(providerId);
     if (stored?.api_key) effectiveApiKey = stored.api_key;
   }

   // Step 2: 再做 NO_CREDENTIALS 校验
   if (!effectiveApiKey && authStyle !== 'env_only') {
     return NextResponse.json({ success: false, error: { code: 'NO_CREDENTIALS', ... } });
   }

   // Step 3: 正常走 testProviderConnection
   ```

**Fix B — 前端编辑态改造（PresetConnectDialog + ProviderForm 两条路径都要改）：**

前端**不能**把 mask 值放进 `apiKey` state。替代方案是**单独维护 `hasStoredKey` 标记**：

1. **打开编辑对话框时** — 不再 `setApiKey(editProvider.api_key || "")`，改成：
   ```ts
   setApiKey('');                                    // 输入框空
   setHasStoredKey(!!editProvider.api_key);          // 记住"DB 里有 key"
   ```
2. **输入框 UI** — 若 `hasStoredKey && !apiKey` 显示占位符 `已保存 ************`（翻译文案）；用户一旦输入就视为修改
3. **测试/保存时** — 构造请求 body 的逻辑改成：
   ```ts
   body: JSON.stringify({
     providerId: isEdit ? editProvider.id : undefined,
     ...(apiKey ? { apiKey } : {}),  // 未修改则完全不带这个字段
     baseUrl,
     ...
   })
   ```
4. **需要同时修改的文件：**
   - `src/components/settings/PresetConnectDialog.tsx:103`（测试 body）
   - `src/components/settings/PresetConnectDialog.tsx:154`（加载编辑态 setApiKey）
   - `src/components/settings/ProviderForm.tsx:98`（加载编辑态 setApiKey）
   - `src/components/settings/ProviderForm.tsx:190`（保存时的 api_key 字段）

**保存路径的附加说明：** 即使前端"未修改则不发 apiKey"，后端 PUT `src/app/api/providers/[id]/route.ts:52-55` 的 mask 防护仍保留（双重保险），`updateProvider()` 的 `data.api_key ?? existing.api_key` 也保留。三层防护谁都别拆。
